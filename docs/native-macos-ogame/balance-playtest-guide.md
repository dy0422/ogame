# 平衡测试指南

本项目采用快节奏单机局，不追求原版服务器的真实等待时间。当前目标窗口：

- 首艘舰船：T+10 到 T+25 分钟。
- 首次舰队出航：T+20 到 T+45 分钟。
- 首次冲突：T+45 到 T+90 分钟。
- 首次胜利：T+2 到 T+4 小时。

## 自动基线

运行：

```bash
swift run OGameBalanceTool
```

输出字段：

- `first_ship`
- `first_fleet`
- `first_espionage`
- `first_exploration`
- `first_conflict`
- `first_colony`
- `first_moon`
- `first_moon_action`
- `victory_at`
- `ai_attacks`
- `automation_actions`
- `player_rank`
- `events`
- `reports`

`OGameBalanceTool` 使用 `BalanceScenarioRunner`，包含轻量玩家引导脚本，用来验证一局新档可以稳定走到造舰、出航、侦察/探索、冲突、殖民和胜利。新增字段用于观察舰队槽、探索事件、月球系统和自动托管是否把节奏推离目标窗口。

同一命令还会追加 `Autoplay gameplay audit` 段落。它使用 `GameplayAuditEngine`，不注入舰船、月球或胜利条件，只依赖舰队优先自动托管、顾问建议和常规模拟推进，用来检查真实开局是否自然产生玩法深度。

审计字段：

- `used_guided_fixtures`：应为 `false`，表示没有使用引导测试夹具。
- `advisor_kinds`：审计窗口内采样到的顾问建议类型。
- `route_progress`：经济、科技、统治、探索四条胜利路线的检查点完成度。
- `route_next`：每条路线的下一项未完成检查点。
- `expansion_signals`：星区事件、PVE 目标、行动链、贸易线、深度情报、遗物和危机的总信号数。
- `commander_signals`：指挥官本体、招募令和训练数据折算后的总信号数，用于确认指挥官模块在真实开局中可见。
- `ai_intents`：AI 当前行为意图，例如扩张、侦察、掠袭、防御或回收。
- `notes`：审计结论，例如真实推进、舰队启动受阻、AI 压力可见、顾问覆盖良好。

审计若出现 `舰队启动受阻`，说明自动托管没有在快节奏窗口内自然造出第一艘船或触发首次出航，应优先检查开局资源、第一艘船预算保护、建筑前置和自动队列策略。

## 手动 Playtest 记录

每次完整手动局记录：

- 第一次升级矿场的时间。
- 第一次造船的时间。
- 第一次侦察或探索的时间。
- 第一次舰队召回的时间。
- 第一次战斗的时间。
- 第一次殖民的时间。
- 第一次月球和第一次月球动作的时间。
- 胜利路线和胜利时间。
- 卡住、看不懂、需要更多提示的界面。
- 保存、读取、离线补算是否可信。

## 调参原则

- 玩家开局不能等太久，10 分钟内应看到明确目标。
- AI 必须主动发展，但离线期间不能连续攻击到让玩家无从挽回。
- 经济胜利不能只靠囤资源过早触发。
- 后期舰船成本应明显高于巡洋舰/战列舰，但在快节奏局中仍可见。
